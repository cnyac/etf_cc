# 《养家·装载契约》导读（写给团队的大白话版）

> 这份文档是给第一次接触《养家·装载契约》的同事看的。它不替代契约本身，而是把契约
> 每一节"到底在说什么、为什么要有它"用最直白的话讲一遍。读完这份，再去看
> `config/养家·装载契约.md` 就不会懵。

---

## 一个寓言：请名厨，装餐盒

先记住一个画面，后面全靠它：

> 我们请了一位**名厨**（炒股养家）来做菜。这位名厨手艺极好，但有自己的脾气——他习惯
> 一边做一边吆喝、最后还要说一句招牌口头禅。
>
> 问题是：我们不是开堂食，而是**做标准化外卖**。每一份菜最后都得装进一个**格子固定的
> 餐盒**（JSON），多一格少一格、装错格子，质检员（Python 校验）当场退货。
>
> 于是中间就缺了一个东西：一张**"装盒说明书"**，告诉名厨"你那道汤舀进哪个格、你那句
> 口头禅贴在哪、你说的'火候'对应盒子上印的哪个标准词"。

**这张"装盒说明书"，就是《养家·装载契约》。**

- **名厨的心法**（他怎么尝味、怎么判断火候）→ 写在另一份文档《炒股养家影子分身》里，
  那是"灵魂"，我们一个字都不改。
- **今天的食材和订单**（这桌客人点了什么、有哪些菜）→ 由 Python 程序每次现给（gen_prompt）。
- **餐盒的格子形状和质检标准** → 写死在代码 `src/llm_schema.py` 里。
- **装盒说明书** → 就是这份契约，把"名厨的发挥"翻译成"装进哪个格子"。

名厨负责好吃，餐盒负责标准，**契约负责让好吃的东西不洒出格子**。

---

## 逐节解释契约在讲什么

### 〇、最高原则——"名厨是主厨，但今天按外卖规矩来"

三句话：

1. **养家是总指挥**：店里还有别的厨师（赵老哥、冯柳），但养家说了算。
2. **再好吃也得装盒**：名厨可以尽情发挥，但发挥完**必须**落到餐盒的格子里，不能只是
   端出一盘散装的菜（散文报告）就走。
3. **别理灵魂文档里的"客套话"**：名厨的心法文档结尾写着"请把菜单给我、我准备好了"这种
   对话腔——那是给"人机聊天"写的。在我们流水线里，食材已经在订单里了，别停下来等人喂，
   直接出成品。

### 一、翻译表——"哪道菜舀进哪个格子"（全文最核心）

这是整份契约的心脏。名厨的自然输出有 5 道"菜"，每道菜在餐盒里都有固定的格子：

| 名厨端出来的（影子分身 5 段） | 装进餐盒哪个格（JSON 字段） | 大白话 |
|---|---|---|
| ① 大局定性（天气/循环/阶段） | `stage` / `market_phase` / `style_tone` 等枚举格 | "今天什么天气、行情在哪个季节"→ 填进固定的几个标准词格子 |
| ② 人心图谱（持筹/持币者心态） | `panorama_text` | "场内场外的人都在想啥"→ 一大段文字 |
| ③ 异动扫描（盯哪几只） | `ticker_analyses` | "今天最值得说的 1-2 只票"→ 按代码逐条点评 |
| ④ 实战指令（拼/试/退） | `style_tone` + 进攻/撤退方向 | "该全力上、还是试试、还是跑"→ 填定调 + 方向 |
| ⑤ 养家心语（招牌口头禅） | `free_analysis` 末尾 | 那句灵魂金句，没有专门的格子，就**贴在自由发挥段的结尾** |

> 为什么要这张表？因为名厨说"今天艳阳高照、该全力拼"，机器听不懂"艳阳高照该放哪"。
> 这张表就是翻译：把名厨的话，对应到餐盒上印好的格子。

### 二、enum 词汇对照——"名厨的土话 ↔ 盒子上的标准词"

有些格子不是随便写的，只能填**白名单里的固定词**（叫 enum）。填了白名单外的词，质检员
直接退货。

名厨爱说土话（"艳阳高照""全力拼取"），这一节就是一张**土话↔标准词对照表**，告诉名厨：
你想说"全力拼"，就在 `style_tone` 这个格子里**原样写"全力拼取"**这四个字，别写成"梭哈"
或"满仓干"——那些不在白名单上。

本轮我们特意把白名单**改成了名厨的土话**（艳阳高照/风暴来袭/全力拼取……），所以名厨说
什么、格子里就填什么，几乎不用拐弯。这是"保住名厨灵魂"的关键一步。

### 三、装载纪律——"每个格子的填写规矩"

这节是细则，回答三类问题：

1. **养家自己的格子有哪些、哪些必填**（`yangjia_emotion_cycle` 那一组）。
2. **养家这次还多管了哪些格子**：原来有个"北京炒家"厨师，现在这位厨师走了，他的活
   （风险点、独特异象、周末宏观、逐只票的纪律审）**全交给养家兼着**。
3. **没菜怎么办**：某个格子今天没东西可装（比如某类票一只都没有），就**填 null（明确写"空"）**，
   并在总结里说一句为什么空——不要硬塞、不要留个空字符串糊弄。
4. **字数规矩**：大段文字只有**下限没有上限**（别偷工、但也别为凑字数注水）；唯独"证据"
   那栏要**精简**（一两句点到关键词和数字就行）。

### 四、失活声明——"把名厨的客套话静音"

名厨的灵魂文档是为"聊天"写的，结尾老有"请给我数据""我准备好了，掌握市场之心……"
这类话。这一节就是一个**静音开关**：明确告诉名厨——在流水线里这些客套话**全部作废**，
不要回那句口头禅确认、不要等人喂数据、不要端散装菜，**只交一份装好盒的成品（JSON）**。

好处：这样我们就**不用去改名厨的灵魂文档**（你满意的那份原封不动），客套话在契约这一层
就被中和掉了。

### 附、字段速查——"餐盒格子一览表"

最后一张表，把养家负责的所有格子列出来，标明类型和归属。它和代码 `src/llm_schema.py`
里的定义必须**一字不差**；一旦对不上，测试会报红（见下面清单里的防漂移测试）。

## 要彻底看懂这份契约，还需要看哪些东西（配套阅读清单）

按"先看什么"分成三梯队。**只想快速上手，看第一梯队就够了。**

### 第一梯队 · 必读（看懂契约的前提）

| 内容 | 在哪 | 为什么要看 |
|---|---|---|
| **名厨的灵魂文档** | `config/炒股养家影子分身：全动态大局观监控插件.md` | 契约是"把这份文档的输出装盒"，不先读它，契约里"5 段输出"无从谈起。**先读这份。** |
| **餐盒格子的真身** | `src/llm_schema.py` | 所有 JSON 字段名、enum 白名单的**唯一源头**。契约里每个字段/枚举词都来自这里。 |
| **人格分工配置** | `config/personas.yaml`（A 段 `yangjia_emotion_cycle`） | 养家"管哪些格子"（`output_fields`）、指向哪份灵魂文档（`skill_ref`）就配在这里。 |

### 第二梯队 · 理解约束与流程（想改东西前必读）

| 内容 | 在哪 | 为什么要看 |
|---|---|---|
| **质检员（校验规则）** | `src/llm_validate.py` | 装错格子为什么被退货、降级机制怎么触发，规则都在这。 |
| **最终 prompt 怎么拼** | `src/llm_prompt.py` | 看契约在整条流水线里站在哪一步（灵魂文档→契约→数据→schema 怎么串起来）。 |
| **全人格大图** | `docs/PERSONAS.md` | 养家不是一个人在战斗，这份讲清四象限 × 各人格的分工矩阵。 |
| **项目铁律与禁区** | `CLAUDE.md`（"绝对禁区""分析撰写规则"两节） | 理解"为什么不能乱造归类、不能乱改 enum、术语为什么不能用错"。 |

### 第三梯队 · 深挖与验证（可选）

### 一句话记忆法

> **灵魂文档** = 名厨怎么想（`影子分身`）；
> **本契约** = 怎么把想的装进盒子（`养家·装载契约`）；
> **schema/校验** = 盒子的格子和质检（`llm_schema` / `llm_validate`）；
> **Python prompt** = 今天的食材订单（`gen_prompt`）。
> 四者各管一段，互不重复。
